# AI 对话应用 技术架构文档

## 1. 整体架构

前后端分离，前端通过 HTTP / SSE 调用后端 REST API。

```
┌─────────────┐        HTTPS / SSE        ┌──────────────────┐        OpenAI 兼容协议        ┌────────────────┐
│   Frontend   │ ───────────────────────▶ │  Backend (FastAPI) │ ───────────────────────────▶ │ 第三方 LLM 服务 │
│  Next.js     │ ◀─────────────────────── │                    │ ◀─────────────────────────── │ (DeepSeek 等)   │
└─────────────┘                           └──────────────────┘                                └────────────────┘
                                                    │
                                                    ▼
                                              ┌──────────┐
                                              │  MySQL    │
                                              └──────────┘
```

## 2. 技术栈

| 层 | 技术 |
| --- | --- |
| 前端框架 | Next.js（App Router）+ TypeScript |
| 前端状态管理 | Zustand |
| 后端框架 | FastAPI + Uvicorn |
| ORM | SQLAlchemy 2.x（Mapped/mapped_column 风格） |
| 数据库驱动 | PyMySQL（MySQL） |
| 鉴权 | JWT（python-jose）+ bcrypt 密码哈希（passlib） |
| LLM 调用 | langchain-openai（`ChatOpenAI`，OpenAI 兼容协议，流式 + 工具调用） |
| 外部工具 | langchain-mcp-adapters + mcp：把 MCP server（如 MySQL MCP）暴露的工具接入 langchain 工具集 |
| 对象存储 | oss2（阿里云 OSS 官方 SDK），封装为 `app/utils/oss.py` 的 `OssClient`，存放 LLM 产出的图片/视频/文件等资源 |
| 向量库 | pymilvus（Milvus 官方 SDK），封装为 `app/utils/milvus.py` 的 `MilvusStore`，提供集合管理 / 检索 / upsert。已用于：对话自动索引（`conversation_index_service`）、知识库文档导入与管理（`knowledge_base_service`，独立 KB 集合）、对话时 RAG 检索（`rag_service`，以 `search_knowledge_base` 工具形式由模型按需调用） |
| 文本向量化 | Qwen3-Embedding（默认 DashScope 兼容模式 `text-embedding-v4`，OpenAI 兼容协议），封装为 `app/llm/embedding.py` 的 `EmbeddingClient`；已用于对话自动索引、知识库导入与对话 RAG 检索 |
| 文档解析 | PDF 用 pymupdf4llm 转 Markdown（表格保留为 markdown 表）+ PyMuPDF 抽图；Word(python-docx)/Excel(openpyxl) 表格也转 markdown，封装在 `app/services/document_parsing/`；配合表格感知切分（`utils/text.split_markdown`），图片交 OSS + OCR |
| 图片 OCR | RapidOCR（rapidocr-onnxruntime），封装为 `app/utils/ocr.py`；较重，未安装时自动降级为不识别图中文字 |
| 重排 rerank | DashScope gte-rerank（HTTP API，httpx），封装为 `app/utils/rerank.py`；对 Milvus 召回结果重排提升质量，默认关闭，失败降级为原顺序 |
| 多模态 VL | Qwen-VL（DashScope，OpenAI 兼容，openai SDK），封装为 `app/llm/vl.py`；可选用于把 PDF 含表页面图片转 markdown 还原复杂表格，默认关闭，失败回退文本提取 |
| 配置管理 | pydantic-settings，统一从 `.env` 读取，禁止硬编码密钥 |

## 3. 后端分层架构

```
app/
├── main.py     # 应用入口：挂载中间件、路由，启动时建表 + 加载 MCP 工具
├── core/       # config（配置）/ security（JWT+密码哈希）/ deps（依赖注入）
├── db/         # engine、SessionLocal、Base
├── models/     # ORM 模型
├── schemas/    # Pydantic 请求/响应模型
├── llm/        # LLM 组件：封装 langchain（client 流式+工具循环 / tools 本地工具 / mcp MCP 加载 / embedding 文本向量化），对外暴露 stream_reply、LlmCallError、load_mcp_tools、EmbeddingClient
├── services/   # 业务逻辑层（chat_service 经 app.llm 调用；含 knowledge_base 导入、rag 检索、document_parsing 多格式解析子包）
├── routers/    # 路由层（含 knowledge_base 文档上传）
└── utils/      # 技术辅助：time / oss / milvus / text（切分）/ ocr（图片识别）/ rerank（重排），不含业务规则
```

### 3.1 分层职责与约束（代码注释中明确声明）
- **routers**：只做参数解析、依赖注入、组装响应，不直接操作 ORM、不写业务规则。
- **services**：承载全部业务规则——鉴权流程、会话归属校验、游客限额判断、LLM 请求拼接与调用。
- **core**：纯技术能力（JWT 编解码、密码哈希、依赖注入获取 DB/当前用户），不包含业务判断。
- **utils**：无业务含义的辅助函数（如时间格式化），业务规则不允许下沉到这一层。

这一约束直接对应 `.claude/rules/architecture.md` 中"不要跨层直接调用内部实现""不要把业务逻辑写进工具函数"的要求，是当前代码库里少数已经被严格执行的架构规则。

### 3.2 鉴权与身份解析（`core/deps.py` + `core/security.py`）
- `OAuth2PasswordBearer(auto_error=False)`：允许匿名请求，由各路由自行决定是否强制登录。
- `get_optional_current_user`：解析 JWT，无 token 或无效 token 时返回 `None`（视为游客），不抛异常。
- `get_current_user`：基于上者，若为 `None` 则抛 401，用于强制登录的路由（sessions、auth/me 等）。
- `get_client_ip` / `get_anonymous_id`：分别从 `X-Forwarded-For` 头和 `X-Anonymous-Id` 头取游客身份线索。
- JWT：`sub` 存用户 ID，有效期 `JWT_EXPIRE_MINUTES`（默认 7 天），密钥/算法均来自配置。

### 3.3 路由与业务服务对照

| 路由 | 前缀 | 是否要求登录 | 对应 service |
| --- | --- | --- | --- |
| `auth.py` | `/api/auth` | 部分（register/login 不需要，me/change-password 需要） | `auth_service` |
| `sessions.py` | `/api/sessions` | 是 | `session_service` / `message_service` |
| `chat.py` | `/api/chat` | 否（区分游客/登录两条路径） | `chat_service` / `guest_quota_service` |
| `knowledge_base.py` | `/api/knowledge-base` | 是 | `knowledge_base_service` |

### 3.4 对话流式架构（`chat_service.stream_chat_reply`）

核心设计约束：**`StreamingResponse` 一旦开始 yield，HTTP 200 已发出，此时不能再 `raise HTTPException`**（否则连接异常中断而非返回正常错误响应）。因此：

- 游客超额、会话不存在、LLM 调用失败等所有错误，统一转换为 SSE 事件帧 `{"type": "error", "error": "..."}` yield 给前端，再正常 `return` 结束流，而不是抛异常。
- SSE 事件类型：`session`（新会话 id）、`delta`（最终回复增量文本）、`reasoning`（思考模式下的思维链增量）、`step`（工具调用执行步骤：`tool`/`phase`(start/end)/`args`，用于展示执行过程）、`error`、`done`。

流程：
1. 判断 `user` 是否为 `None`：
   - 登录：解析/创建会话（`_resolve_session_for_user`），用户消息先落库，再 yield `session` 事件。
   - 游客：通过 `guest_quota_service.resolve_guest_identity` + `check_and_increment_quota` 校验每日额度（超额抛 429，被外层转为 SSE error）。
2. **RAG 以工具形式接入（`rag_service.make_kb_search_tool`）**：按 `kb_scope`（`none/global/personal/both`，默认 `both`）与 `user_id` 构造一个绑定上下文的 `search_knowledge_base` 工具，交给模型在「需要时」自行调用（function calling），而非每轮强制召回拼接。范围不可检索（`none`、或游客选 `personal`）或知识库未配置时不挂该工具（返回 None）；挂载时给 system prompt 追加一句使用提示。构造工具本身无 IO，真正的检索（问题→向量→Milvus 过滤召回→可选 rerank→拼接）在模型调用工具时经工具循环执行，失败降级为「未找到相关内容」，不影响回复。
   - **rerank 重排（`app/utils/rerank.py`，可选）**：`RERANK_ENABLED=true` 时，先从 Milvus 粗召回 `RAG_RECALL_K`（默认 20）条候选，再用交叉编码器（默认 DashScope `gte-rerank-v2`，HTTP API）按与问题的相关性重排，取前 `RAG_TOP_K`（默认 5）拼进上下文——向量检索偏召回、rerank 偏精确，组合提升进入上下文的片段质量。未启用 / 未配置 / 调用失败时降级为保持向量检索原顺序，不影响检索。rerank 凭证默认复用 `EMBEDDING_API_KEY`。
3. `_build_context_messages`：拼接 system prompt（挂知识库工具时附使用提示）+ 历史消息（仅登录会话有历史）+ 当前用户输入。
4. `app.llm.stream_reply`（`app/llm/client.py`）：`chat_service` 经 `app.llm` 组件的这一接口调用，不感知 langchain 细节。内部用 langchain-openai 的 `ChatOpenAI`（`bind_tools` 后）流式调用第三方模型，逐段产出 `(kind, text)`——`kind="reasoning"`（来自 chunk `additional_kwargs.reasoning_content`，思考模式专有）转为 SSE `reasoning` 事件，`kind="content"`（来自 `chunk.content`）转为 SSE `delta` 事件，`kind="tool"`（工具调用的 start/end）转为 SSE `step` 事件透传给前端展示执行过程；调用异常统一包装为对外的 `LlmCallError`。
   - **工具调用**：模型可请求三类工具——本地工具（`app/llm/tools.py` 的 `LOCAL_TOOLS`，当前为 `get_current_date`）、启动时加载的 MCP 工具、以及调用方按请求经 `stream_reply(messages, extra_tools=...)` 注入的工具（如绑定了用户/范围的知识库检索工具 `search_knowledge_base`）。某轮流式输出携带 `tool_calls` 时，在服务端执行工具（统一 `await ainvoke`，兼容本地/MCP/注入工具）、把 `AIMessage(tool_calls)` 与 `ToolMessage(result)` 回灌后发起下一轮，直到模型给出不含工具调用的最终回复（最多 `_MAX_TOOL_ROUNDS=5` 轮）；工具中间过程不产出 `content`、不落库。langchain 工具的封装统一收敛在 `app/llm`（`build_tool` 把普通可调用对象包成 langchain 工具，供 service 层构造带上下文的工具而无需依赖 langchain）。
   - **MCP 工具（`app/llm/mcp.py`）**：应用启动时（`main.py` 的 `load_external_tools`）调用一次 `load_mcp_tools()`，经 `langchain-mcp-adapters` 的 `MultiServerMCPClient` 以 stdio 子进程拉起配置的 MySQL MCP server 并加载其工具，缓存到模块级（不在每次请求时重复拉起进程）。MySQL 连接参数从 `MYSQL_MCP_DATABASE_URL`（留空则回退 `DATABASE_URL`）解析后经环境变量（`MYSQL_HOST/PORT/USER/PASSWORD/DATABASE`）传给子进程，子进程 env 继承当前进程环境以保留 `PATH` 等。默认关闭（`MYSQL_MCP_ENABLED=false`）；未开启或加载失败都降级为"无 MCP 工具"，不阻断启动。当前默认 server 为 `mysql-mcp-server`，暴露 `execute_sql` / `get_schema_info` / `get_table_sample` 三个工具。⚠️ 开启后 LLM 可经 MCP 对数据库执行 SQL，务必用 `MYSQL_MCP_DATABASE_URL` 指向只读账号 / 独立库，不要直连存有密码哈希的主库。
5. 登录用户在流结束后，将完整 assistant 回复（仅 `content` 部分）落库；思维链不落库、不回灌上下文；游客模式不落库。
   - **内联资源转存 OSS（`app/services/resource_service.py`）**：落库前把回复中内联的 base64 资源（`data:<mime>;base64,...`，模型产出图片/视频/文件的唯一内联形式）转存阿里云 OSS、替换为外链，避免单条历史消息膨胀到 MB 级。仅作用于落库的登录路径（游客不落库、不上传）；`oss2` 为阻塞 IO，经 `asyncio.to_thread` 放到线程池执行，不阻塞流式 generator 的事件循环。未配置 OSS（`OSS_*` 留空）或单个资源上传失败时降级——保留原始内联、不中断主流程。实时流式 `delta` 仍发送原始文本，前端可即时渲染。
   - **对话自动索引进 Milvus（`app/services/conversation_index_service.py`）**：assistant 回复落库后，把本轮「用户提问 + AI 回复」经 `EmbeddingClient`（Qwen3-Embedding）向量化，写入 Milvus，主键复用 `chat_messages.id`（便于检索命中后映射回原消息），同时存 `text/role/session_id` 等字段。仅登录路径（游客不落库、不索引）；embedding/milvus 均为阻塞 IO，经 `asyncio.to_thread` 放线程池。集合进程内只 `ensure_collection` 一次。`EMBEDDING_*` 或 `MILVUS_*` 未配置、或任一步失败时整体降级（仅记日志、跳过），绝不影响本次回复——索引是检索（RAG）的旁路收益。

### 3.5 游客限额（`guest_quota_service`）
- 维度：`(identity, usage_date)` 唯一约束（`uq_guest_identity_date`），identity 优先取前端 `anonymous_id`，否则回退 `ip:{client_ip}`。
- 每次请求先查当日记录，不存在则创建（count=0），达到 `GUEST_DAILY_LIMIT`（默认 5）则抛 429，否则 count+1 并提交。
- 时间基准用 `today_utc_date()`（`utils/time.py`），避免与服务器本地时区耦合。

### 3.5.1 知识库文档管理（`knowledge_base_service`）
- 入口（均要求登录）：一步导入 `POST /api/knowledge-base/documents`、预览 `POST /documents/preview`、提交 `POST /documents/commit`、查看切片 `GET /documents/{doc_id}/chunks`、列出 `GET /documents`、删除 `DELETE /documents/{doc_id}`。路由只解析参数、把阻塞流程放 `asyncio.to_thread`、把异常按类型转 HTTP（400/403/404，统一经 `_to_http_error`）。
- **导入流程**：**解析（`document_parsing.parse_document`，按扩展名分派）→ 切分（`utils/text.split_text`，`KB_CHUNK_SIZE/OVERLAP`）→ 向量化（`EmbeddingClient` 批量）→ 写入 Milvus 知识库集合（`upsert`）**。`scope` 取 `global`（全局共享）或 `personal`（仅自己，默认），`owner_id` 记录上传者。
- **预览 → 编辑 → 提交（入库前可微调）**：`preview_document` 只解析+切分、返回 chunk 列表（含图片在此阶段已转存 OSS+OCR 的最终文本），**不写 Milvus**；前端逐段编辑后调 `commit_chunks` 才向量化入库。`import_document` = `preview_document` + `commit_chunks` 的一步封装。已入库文档可经 `get_document_chunks` 取回分片再次预览/编辑，编辑后 `commit_chunks` 覆盖（同 `doc_id`）。`commit_chunks` 按 `doc_id` **先删旧分片再写新**（幂等，支持重导/编辑覆盖），覆盖时校验归属（仅上传者，否则 403），并丢弃空白 chunk。
- **支持格式与多模态内容**：`.txt/.md/.pdf/.docx/.xlsx`（`.doc/.xls` 暂不支持，返回明确指引）。解析层把文件拆成有序内容块：文本块、表格（序列化为 markdown 表）、图片块（保留原始字节）。**PDF 用 pymupdf4llm 转 Markdown**（表格保留为 markdown 表，避免 `get_text` 把表揉烂、且不再与单独抽表重复）。注意：pymupdf4llm 新版 layout 模式会自动调 RapidOCR，与本机 rapidocr 版本 API 不兼容会崩溃并退回 `get_text`，故 `parse_pdf` 显式 `use_layout(False)` 走经典提取器。**复杂表格（合并表头等）可选开启多模态 VL**（`PDF_VL_ENABLED=true`）：`parse_pdf` 注入 `app/llm/vl.image_to_markdown` 回调，对**检测到表格的页面**渲染成图片交 Qwen-VL 转 markdown（更好还原合并表头），其余页面仍走经典提取；VL 调用经回调注入，解析层不依赖具体模型、保持可单测；VL 失败该页回退文本提取。默认关闭（每含表页一次 VL 调用，慢且耗 token）。
- **表格感知切分（`utils/text.split_markdown`）**：文本块切分时把连续的 markdown 表格识别为整体——小表占一个 chunk、**绝不跨 chunk 截断**；超长表按数据行分块并**每块重复表头**（含分隔行），保证每块都能读懂列含义；无表格时退化为按字符的 `split_text`。这解决了「复杂表格被切碎导致任一 chunk 都拿不到完整表」的问题。每张图片由 `knowledge_base_service` 上传 OSS 取链接 + `utils/ocr.ocr_image`（RapidOCR）识别图中文字，合成「`[图片] 地址:... 图中文字:...`」文本块入库；OSS 未配置 / OCR 不可用各自降级，两者皆空则跳过该图。每条记录附带 `kind`（text/image）与 `image_url` 字段。解析层只抽取结构、不做 OSS/OCR I/O，保持可单测。
- **存储**：写入 `MILVUS_KB_COLLECTION`（与对话索引集合分开，id 空间独立）。每次上传生成新 `doc_id`（uuid），分片主键 `_chunk_id(doc_id, index)` 由 `sha256` 派生为稳定 int64，故同文档重导可 `upsert` 覆盖；字段含 `text/doc_id/filename/chunk_index/scope/owner_id`。
- **可见性与权限**（无管理员角色，规则力求简单安全）：列出 = 全部 `global` + 自己的 `personal`（过滤表达式 `scope == "global" or owner_id == {me}`，再按 `doc_id` 聚合统计片数）；删除 = **仅上传者本人**（按 `doc_id` 查 `owner_id` 校验，他人→403、不存在→404）。`doc_id` 用 32 位 hex 正则校验，兼防过滤表达式注入。
- 与「对话自动索引」的差异：本流程是**用户主动发起**且有返回，失败**显式报错**（未配置 / 非 UTF-8 / 空内容 / 向量化或写库出错 / 无权 / 不存在），不静默降级。
- 检索增强（RAG）已接入对话：见 §3.4 流程第 2、3 步与 `rag_service`。

### 3.6 数据模型与关系

```
User (1) ──< ChatSession (1) ──< ChatMessage
GuestUsage  (与 User 无关联，独立按 identity+date 计数)
```

- `users`：`username`（唯一）、`password_hash`。
- `chat_sessions`：`user_id` FK（`ondelete=CASCADE`）、`title`，`created_at`/`updated_at` 由数据库 `func.now()` 维护。
- `chat_messages`：`session_id` FK（`ondelete=CASCADE`）、`role`（user/assistant）、`content`（Text）。
- 会话/消息删除通过 ORM `cascade="all, delete-orphan"` 级联清理，未引入额外的应用层清理逻辑。
- 当前为开发期自动建表（`Base.metadata.create_all`），未引入 Alembic 迁移。

```
└── types/          # auth/session/message/knowledgeBase 类型定义
```

### 4.1 鉴权头与游客标识统一构造（`lib/api/client.ts`）
- 所有请求统一通过 `apiFetch` 发出；若本地存在 token 则附加 `Authorization: Bearer`，否则附加 `X-Anonymous-Id`（由 `anonymousTrial` 生成并持久化）。
- SSE 流式请求复用同一套 header 构造逻辑（`buildStreamHeaders`），避免鉴权逻辑重复实现。
- 上传文件时（`body` 为 `FormData`）不强设 `Content-Type`，交由浏览器带 multipart 边界；错误消息兼容 FastAPI 的 `detail` 与历史 `message` 两种字段。

### 4.2 SSE 流式消费（`lib/api/chat.ts` + `store/useChatStore.ts`）
- 前端手动按 `\n\n` 分隔 SSE 事件，兼容一次 `read()` 拿到多条或半条事件的情况，剩余半条缓存到下次拼接（`parseSseBuffer`）。
- `useChatStore.sendMessage`：
  - 未登录且本地试用次数已耗尽时直接拦截，置 `guestLimitReached`，不发请求（前端先挡一层，后端 `guest_quota_service` 再挡一层，双重保护）。
  - 乐观更新：先在本地插入用户消息 + 占位的 streaming assistant 消息，再随 `onDelta` 增量追加最终回复内容；思考模式下 `onReasoning` 增量追加到消息的 `reasoning` 字段，由 `MessageBubble` 以可折叠的「💭 思考过程」区块展示。
  - 执行过程：`onStep`（`step` 事件）把工具调用维护成消息的 `steps` 列表（start 追加一条「执行中」、end 标记完成），`MessageBubble` 以可折叠的「🛠 执行过程」时间线展示（如「检索知识库：xxx」「执行数据库查询」），转圈→✓。`reasoning`/`steps` 均不持久化，刷新或切回历史会话后只剩最终回复。
  - 新会话场景：发送时 `sessionId` 为空，消息先挂在 `"__draft__"` key 下；收到后端 `onSessionId` 后，用真实 session id 重新挂载消息列表。
  - 支持 `AbortController` 中断流（"停止生成"），中断被视为正常结束而非错误。

### 4.3 状态管理边界
- `useAuthStore`：当前用户、登录态。
- `useSessionStore`：会话列表、当前选中会话。
- `useChatStore`：消息内容、流式状态、游客限额提示、错误信息；通过读取 `useAuthStore`/写入 `useSessionStore` 实现跨 store 协作，未引入额外的全局状态库。

### 4.4 知识库页面与 RAG 检索范围（`app/knowledge-base` + `lib/api/knowledgeBase.ts`）
- 知识库管理页（需登录，侧边栏入口）：上传文档（.txt/.md/.pdf/.docx/.xlsx + 可见范围 global/personal）→ **预览切片**（逐段展示，可编辑文本/删除某段）→ **确认导入**；列出可见文档（全局 + 自己的个人，含分片数与范围徽标）；对已入库文档可「**查看切片**」（取回分片复用同一 `ChunkEditor`，编辑后「保存覆盖」重新入库）；删除自己上传的文档（`ConfirmDialog` 二次确认；他人文档不显示删除入口，后端兜底 403/404）。预览/提交/查看切片对应 `lib/api/knowledgeBase.ts` 的 `previewDocument`/`commitChunks`/`getDocumentChunks`。页面用本地 `useState` 管理，未引入额外 store。
- 对话检索范围：`ChatInput` 对登录用户展示「知识库检索」下拉（`none/global/personal/both`，默认 `both`），随消息透传到 `ChatStreamPayload.kb_scope`；省略时后端默认 `both`（游客退化为仅全局）。

## 7. 当前已知的架构缺口

- 无数据库迁移工具（Alembic），生产环境的 schema 变更需要手动管理。
- 无限流/熔断/重试机制应对第三方 LLM 服务异常（仅做了错误转 SSE event，未做重试）。
- 自动化测试覆盖 `app/llm` 组件（含 embedding）、`app/utils`（oss / milvus / text / ocr / rerank）、`app/services`（resource / conversation_index / knowledge_base / rag / document_parsing）（pytest，见 `backend/tests/` 与 README「测试」章节）；另有默认跳过的 e2e 用例（`RUN_E2E=1` 开启）。`chat_service` 编排层、路由层尚无测试。
- `GuestUsage` 表无过期清理机制，长期运行会持续累积历史行。
- 知识库相关：仅支持 `.txt/.md/.pdf/.docx/.xlsx`（`.doc/.xls` 未支持）；列出文档为单次 `query`（上限 16384 行）后内存聚合，大规模知识库需改分页/聚合；图片块的检索价值依赖 OCR 命中（OCR 引擎较重，未安装时降级为仅存链接）。极复杂表格（扫描件、大量合并单元格）pymupdf4llm 仍可能识别不全，需要时可再上 unstructured / 表格结构模型或视觉解析。
- RAG 检索仅召回知识库集合，未纳入对话历史集合；未做相似度阈值过滤（只取 top-k）。
